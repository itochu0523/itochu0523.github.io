DMC 商品マスター整備
システム仕様書

ブランドマスター構築・Lazuli カテゴリー標準化・商品マッピングの 3 フローを統合したシステム全体仕様。Claude Code によるシステム化の下地として確定させる。

フロー 1：ブランド — 仕様確定 フロー 2：Lazuli カテゴリー — 仕様確定 フロー 3：商品マッピング — 設計確定 Claude Code 移行：次フェーズ

🎯 システム全体ゴール

すべての商品（JAN コード）に対して、メーカー・ブランド・Lazuli 標準カテゴリーを正確にマッピングし、
顧客・EC プラットフォームごとに必要なカテゴリーを即座に引き出せる状態を作る。

ブランドリストは Claude / Gemini / ChatGPT の 3 LLM で最適化。
カテゴリーは Amazon・楽天・AEON・量販店・ドラッグストア等の各カテゴリーを Lazuli 標準カテゴリーに集約し、常にアップデートし続ける。

フロー 1

ブランドマスター構築

GS1 メーカーリスト → 3 LLM でブランドリスト生成 → 統合・正規化 → maker_brand_master

フロー 2

Lazuli カテゴリー構築

各 EC・量販店カテゴリー + 森井マスタ → LLM 平準化 → Lazuli 標準カテゴリー（L1〜L5）

フロー 3

商品マッピング

JAN × maker_brand_master × Lazuli カテゴリー → 商品マッピングマスター完成

3 LLM の役割分担

Claude（Anthropic）

日本語ブランド情報の精度が高い。コスト最適化（Haiku + Prompt Caching）で大量バッチ処理向き。信頼度スコアを JSON で返す。

ブランドリスト統合・正規化ロジック

1

LLM 3社 並列バッチ投入

メーカーリストを一括投入

GS1 メーカーリストを N 件ずつのチャンクに分割し、3 LLM に同一プロンプトで並列投入。出力は統一 JSON スキーマで受け取る。
{"maker_code":"xxx","maker_name":"xxx","brands":[{"name":"xxx","site_url":"xxx","confidence":"high"}]}

2

多数決スコアリング

3 LLM の結果を突合して信頼度を決定

3社一致：confidence = HIGH → 自動採用
2社一致：confidence = MEDIUM → 自動採用 + レビューフラグ
1社のみ：confidence = LOW → DMC 手動確認キュー
全社不一致：UNKNOWN → 手動確認必須

3

brand_name 正規化ルール適用

表記揺れを統一して格納

① メーカー名を除去（「キリン一番搾り」→「一番搾り」）
② 英語ブランドは英語表記を正とする（BOSS, iMUSE）
③ サブブランドは L1 ブランドに統一（「生茶 ほうじ煎茶」→「生茶」）
④ 重複排除（「オールフリー」「All-Free」→「オールフリー」に統一）

4

maker_brand_master へ書き込み

差分のみ UPDATE / INSERT

confidence = HIGH / MEDIUM を自動挿入。LOW / UNKNOWN は review_queue テーブルに保留。DMC 承認後にマスターへ昇格。

Lazuli 標準カテゴリー — データモデル設計

Lazuli カテゴリーは L1〜L5 の階層構造を持ち、各ノードに各ベンダー・EC のカテゴリーを紐づける。 顧客や用途に応じて、必要なマッピングを即座に引き出せる。

💡 欲しいカテゴリーを欲しい時に引き出すクエリイメージ

-- 商品の Lazuli カテゴリーから Amazon カテゴリーを取得
SELECT vm.vendor_category_name
FROM product_mapping_master pm
JOIN category_vendor_mapping vm
  ON pm.lazuli_category_code = vm.lazuli_category_code
WHERE pm.jan_code = '4901085123456'
  AND vm.vendor_type = 'amazon';

-- 顧客ごとに必要なカテゴリーを切り替えるだけ
-- vendor_type = 'rakuten' / 'aeon' / 'matsukiyo' / ...

Lazuli カテゴリー — 継続アップデートフロー

A

各ベンダーカテゴリーの定期収集

Amazon・楽天・AEON 等のカテゴリーツリーを定期（月次推奨）でクローリング or API 取得。変更差分を検出する。

B

LLM による Lazuli カテゴリーへのマッピング提案

新規・変更カテゴリーを LLM に投入し、既存 Lazuli カテゴリーへの対応を提案させる。既存 L5 に合致するなら紐づけ更新、新規 L5 が必要なら追加候補として出力。

C

承認フロー → マスター反映

HIGH 信頼度は自動反映。MEDIUM / LOW は DMC レビュー後に承認。Lazuli カテゴリーが追加された場合は全ベンダーマッピングを自動補完提案。

D

商品マッピングへの自動反映

Lazuli カテゴリー更新後、対象商品のカテゴリー再マッピングを差分実行。常に最新状態を維持する。

継続アップデート・精度向上

Gemini / ChatGPT の追加統合。カテゴリーの継続アップデートフロー稼働。DMC レビューキューの UI 実装。精度モニタリングの自動化。

⚠️ Claude Code 移行前に確定が必要な事項

① GCP 認証・BQ プロジェクト権限：grandline-dev / dmc-ops 両プロジェクトへの書き込み権限
② LLM API キー管理：Claude / Gemini / ChatGPT の API キーを Secret Manager で管理
③ 各ベンダーカテゴリーの取得方法：Amazon / 楽天は公式 API or クローリングのどちらか
④ DMC レビューキューの UI：Spreadsheet で代替するか、専用 UI を作るか
⑤ バッチ実行基盤：Cloud Run / Cloud Scheduler / Airflow のどれを使うか
