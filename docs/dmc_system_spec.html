データを食わせるほど精度とボリュームが上がる自己強化型マスターを構築する。

「AI生成」ではなく「DMCが丁寧に作り上げた資産」としてブランド化（農家の野菜戦略）。
顧客・EC・業界ごとに必要なカテゴリーを即座に引き出せる構造を実現する。
下流案件（商品名+JANのみ）でもマスターから正確な属性を補完できる状態を目指す。

💡 顧客・用途別カテゴリーを即時引き出すクエリ

-- 商品のLazuliカテゴリーから任意の媒体カテゴリーを取得
SELECT vm.vendor_type, vm.vendor_category_name
FROM product_mapping_master pm
JOIN category_vendor_mapping vm
  ON pm.lazuli_category_code = vm.lazuli_category_code
WHERE pm.jan_code = '4901085123456'
  AND vm.vendor_type IN ('amazon','rakuten','aeon');

-- vendor_type を変えるだけで楽天/AEON/量販店カテゴリーを切り替え可能

上流案件 vs 下流案件 — マスターの使い方の違い（MTGで確認済み）

上流案件（登録系・メーカー起点）

メーカーからデータを受領。PDF・Excelを構造化。メーカー名・ブランド名は元データに含まれるためマスター参照は補完的。

マスター：原則不要

下流案件（分析系・POS・メルカリ等）

商品名とJANコードしか残っていない。マスターで補完し精度を高めることが必須。ここがマスターの最大の価値。

マスター：必須

NEW 全体方針：3本柱（6/11確定）

① 差別化と運用課題

カテゴリーマスター運用体制を顧客に見せること自体が競合優位性。現行プロダクトの機能制約で手動作業が残る点が課題。

② 機能統合・資産化

属人的な手動マッピングを廃止。ClaudeやLLMで培ったノウハウを製品機能として統合し、会社の資産として蓄積する。

③ 標準化と将来展開

属性マッピングの標準化とECサイトごとのカラム構造調査を通じ、汎用デモ環境を構築する。
