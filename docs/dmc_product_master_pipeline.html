🗺 全体ゴールと構造

最終ゴール

商品（JAN コード + 商品名）に対して、メーカー・ブランド・カテゴリーを正確にマッピングした
「商品マッピングマスター」を構築・継続的に精度向上させる

各属性は独立したフローで整備し、最終的に 1 テーブルに統合する。仕様を一つ一つ確定させながら進める。

フロー A

商品 × メーカー・ブランド

GS1 データを起点に maker_brand_master を構築。JAN コードの先頭 XX でメーカーを特定し、LLM で商品名からブランドを判定する。

フロー B

商品 × カテゴリー

森井カテゴリマスタ（L1〜L5）を整備し、商品（JAN コード + 商品名）に対してカテゴリーをマッピングする。カテゴリーは継続的に追加・精度向上させていく。

統合

商品マッピングマスター

フロー A + フロー B を統合し、商品 1 件ごとにメーカー・ブランド・カテゴリーがすべて揃った状態を実現する。

dmc-ops.master — ブランド情報取得

メーカー（事業者）1 件 = ブランドサイト 1 件を目標に取得

クローリングは極力行わず、LLM（Claude）へのプロンプトによるブランドリスト生成を主軸とする。ブランドサイトの特定と、そこからのブランド名生成を LLM に委ねる。

ブランドサイト取得

LLM に「このメーカーのブランドサイト候補を出して」と質問。
目標：メーカー 1 件 → サイト 1〜2 件

ブランド名生成

ブランドサイトが判明したら LLM にブランドリストを生成させる。結果を DMC でチェック・精度確認。

💡 Claude コスト最適化 — プロンプト設計のポイント

①
バッチ処理（一問多答）：1 リクエストに複数メーカーをまとめて質問。入力トークンを 50〜70% 削減。
「以下のメーカーに対しブランドサイト URL とブランド名を JSON で返してください。\n{メーカーリスト}」
②
出力フォーマット指定（JSON のみ）：自然文を禁止し出力トークンを最小化。
[{"maker":"xxx","brand_site":"https://...","brands":["A","B"]}]
③
Haiku モデルを優先使用：構造化タスクは claude-haiku-4-5 で十分。Sonnet 比 約 1/5 のコスト。精度不足時のみ Sonnet にフォールバック。
④
Prompt Caching 活用：共通システムプロンプトをキャッシュに乗せると入力コストを 最大 90% 削減。
⑤
2 段階処理：① Haiku で大量生成 → ② 信頼度スコアが低いものだけ Sonnet で再確認。全件を高コストモデルに流さない。

⚠️ 未決定

・1 メーカーあたり何サイトまで許容するか（1 サイト固定 or 最大 2 サイト）
・LLM 生成結果の品質チェックを DMC 手動にするか自動バリデーションにするか
・ブランドサイトの URL 正規化ルール（サブドメイン含むかどうか等）

dmc-ops.master — 非正規化テーブル

maker_brand_master を作成（brand_name 正規化済み）

PK は maker_code 単体ではなく、(maker_code, brand_name) の複合ユニークキー。brand_name は表記揺れを統一・正規化した上で格納する。(maker_name, brand_name) を UK② 候補として併用も検討。

📌 設計上の重要ポイント

maker_code と JAN コードだけではブランドは一意にならない。同一 maker_code でも商品名から複数ブランドが存在する（例：45454545 → 「サントリー伊右衛門」「サントリーBOSS」）。
→ JAN コード + 商品名 を LLM に渡してブランド名を推察・一意化する処理が必要。

dmc-ops.master.maker_brand_master
UK①maker_codeSTRING≒ standard_gs1_company_code
UK①②brand_nameSTRING正規化済みブランド名（必須）
UK②maker_nameSTRINGUK② 候補 (maker_name, brand_name)
brand_site_urlSTRINGLLM 生成 or 手動登録
crawl_attributesJSON拡張属性（最小限）
created_at / updated_atTIMESTAMP
UK①: (maker_code, brand_name) UK②候補: (maker_name, brand_name) brand_site_url crawl_attributes (JSON)

重要 カテゴリーマッピングのロジックと精度向上方針

データモデル（ツリー）：商品(JAN) → 森井カテゴリ L5（最小分類） に1点紐づく。 L5→L4→L3→L2→L1 と一意に遡れ、森井L1でECサイトと接続する。 ECカテゴリは森井L1経由の派生ビューであり、「基本はこれ・Amazonではこれ・楽天ではこれ」と切替表示する。

① 商品 → 森井カテゴリ（精度: 高）

LLM 2段階。Stage1: 商品名＋raw属性（容量/規格等）＋メーカー/ブランド → L1判定。Stage2: L1配下のL4/L5候補（最大1200件）→ L5コード確定。

② 森井L1 → ECカテゴリ（精度: 粗い→改善中）

現状L1単位のため、食品の商品は全てAmazon「食品・飲料・お酒」を表示していた（"ツナがお酒"問題）。画面側で森井L5〜L2ワードとECサブカテゴリを突合し最も近い分類を表示するよう改善。

精度を上げる具体策（ロードマップ）

即効: raw属性ヒント活用 / 候補1200件確保 / ECサブカテゴリ絞り込み（実装済）
中期（本丸）: ④ category_vendor_mapping をL2/L4粒度で構築（L1のみ→中分類レベルへ精緻化） ／ ⑤ 3 LLM多数決（Claude/Gemini/ChatGPT一致でHIGH確定、不一致はレビュー行き） ／ ⑥ 人手修正をfew-shot例として蓄積するフィードバックループ
訂正方法: 商品はPATCHで森井カテゴリ修正（manual化）。EC紐付けはEC調査画面で削除→正しいL1で再追加、または割当バッチ再実行（差分のみ）。

透明性 マスターデータ ダウンロード（関係性の可視化）

データの紐付けを誰でも検証できるよう、関係性ごとに色付きExcelでダウンロード可能。

  • メーカー×ブランド（/makers）: GS1メーカー＋紐づくブランド一覧、confidence色分け
  • 基本×ECカテゴリ（/categories）: 森井L1 × 選択したECサイトの対応カテゴリを横並び。詳細シートにL2サブ・出典（人手/🤖AI）。ECサイトはチェックボックスで選択可
  • 商品統合（/products）: 商品×森井×EC×raw属性
運用 大量取込のバックグラウンド処理＋ジョブ監視

9万件級のCSVも画面と処理を切り分けてタイムアウトを回避。アップロード後すぐ応答を返し、 裏で5000件ずつチャンク処理。進捗は sync_log に逐次反映され、 ダッシュボードのジョブ監視で進捗バー＋件数をリアルタイム表示（実行中は3秒更新）。

Cloud Run: 2Gi / 2CPU / timeout 900s / CPU常時ON / min-instances=1（背景処理がスケールインで中断されない）
