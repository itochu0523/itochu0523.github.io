dmc-ops.master — ブランド情報取得

メーカー（事業者）に対してブランドサイトを 1 対 1 で取得

極力 メーカー 1 件 = ブランドサイト 1 件 を目標とする。クローリングは極力行わず、LLM（Claude）へのプロンプトによるブランドリスト生成を主軸とする。

ブランドサイト取得

LLM に「このメーカーのブランドサイト候補を出して」と質問。
目標：メーカー 1 件 → サイト 1〜2 件

ブランド名生成

ブランドサイトが特定できたら LLM にブランドリストを生成させる。
結果を DMC でチェック・精度確認。

💡 Claude コスト最適化 — プロンプト設計のポイント

大量のメーカーを一括処理する際は、以下の設計でトークンコストを大幅に削減できる。

①
バッチ処理（一問多答）：1 リクエストに複数メーカーをまとめて質問する。
例：「以下のメーカーリストに対して、ブランドサイト URL とブランド名を JSON 形式で返してください。\n{メーカーリスト}」
→ 1 件ずつ質問するより 入力トークンを 50〜70% 削減できる。
②
出力フォーマット指定（JSON のみ）：自然文ではなく JSON のみ返すよう指示することで、出力トークンを最小化する。
例：「余分な説明は不要。JSON 配列のみ返してください。
[{"maker":"xxx","brand_site":"https://...","brands":["A","B"]}]」
③
Haiku モデルを使う：ブランドリスト生成のような構造化タスクは claude-haiku-4-5 で十分な精度が出るケースが多い。Sonnet 比で約 1/5 のコスト。精度が不十分な場合のみ Sonnet にフォールバック。
④
キャッシュ活用（Prompt Caching）：システムプロンプトや共通の指示部分を Anthropic の Prompt Cache に乗せると、繰り返し処理で 入力コストを最大 90% 削減。
⑤
2 段階処理：① まず Haiku でブランドサイト候補を大量生成 → ② 信頼度スコアが低いものだけ Sonnet で再確認。全件を高コストモデルに流さない。

⚠️ 未決定

・1 メーカーあたり何サイトまで許容するか（1 サイト固定 or 最大 2 サイト）
・LLM 生成結果の品質チェックを DMC 手動にするか自動バリデーションにするか
・ブランドサイトの URL 正規化ルール（サブドメイン含むかどうか等）

カテゴリーフロー — 現在設計中

カテゴリーはメーカー・ブランドとは独立したフローとして設計する。
森井カテゴリマスタ（L1〜L5 階層、計 2,735 件）を活用し、商品に対して最小分類（L5）を付与することが目標。マッピングロジックは現在検討中。
